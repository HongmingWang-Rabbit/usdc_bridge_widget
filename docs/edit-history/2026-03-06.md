## Phase 1: Bridge Persistence, Recovery, and Progress Notifications

### Problem Statement

Users were reporting "stolen" USDC because the bridge widget had no persistence. When CCTP burns USDC on the source chain, the user must stay on the page for ~15 min (attestation) and confirm a destination chain mint transaction. If they close the tab, lose connection, or get scared by a false-positive security warning, USDC is burned but never minted. There was no way to resume.

### Solution

Leverages Circle's Bridge Kit `kit.retry(result, context)` method which resumes from the last completed step given a `BridgeResult` object. The solution persists this object to localStorage during bridging and provides a recovery UI on the next visit.

### New Files

#### `src/storage.ts` - localStorage persistence (CRUD)
- SSR-safe with `typeof window === "undefined"` guard
- Key format: `usdc_bridge_tx_{id}` with index at `usdc_bridge_index`
- `PendingBridgeRecord` interface: id, walletAddress, sourceChainId, destChainId, amount, bridgeResult, timestamps, status
- Functions: `savePendingBridge`, `updatePendingBridge`, `loadPendingBridges`, `removePendingBridge`, `cleanupStaleBridges`
- Max 50 records (oldest evicted), stale cleanup after 7 days
- Batched index updates in cleanup (O(n) instead of O(n^2))

#### `src/useRecovery.ts` - Recovery hook
- `useRecovery(options)` returns `{ pendingBridges, retryBridge, dismissBridge, isRecovering, refresh }`
- Loads pending bridges on mount when wallet connected
- `retryBridge(id)`: gets connector -> creates adapter -> `kit.retry(record.bridgeResult, { from: adapter, to: adapter })`
- `dismissBridge(id)`: removes record from localStorage
- Callbacks: `onRecoveryComplete`, `onRecoveryError`, `onPendingBridgeDetected`

#### `src/__tests__/storage.test.ts` - Storage tests
- 16 tests covering all CRUD operations, cleanup, max record eviction, and edge cases

### Modified Files

#### `src/useBridge.ts` - Persistence integration
- Added `BridgePersistenceConfig` interface: `{ walletAddress, sourceChainId, destChainId }`
- Added `pendingRecordIdRef` for tracking current persistence record
- Added `persistenceRef` to avoid object identity issues in useCallback deps
- `handleBurn`: Creates `PendingBridgeRecord` with partial BridgeResult (point of no return)
- `handleFetchAttestation`/`handleMint`: Progressively update `bridgeResult.steps` in persisted record
- On success: removes persisted record
- On error: updates record to `"recovery-pending"`, extracts BridgeResult from error when available
- BridgeChain resolution moved before event handlers (was after, relied on temporal dead zone)

#### `src/BridgeWidget.tsx` - UI additions

  - Tracks `lastActiveStep` via ref to show which step failed on error
- **RecoveryBanner**: Shows at top of widget when `pendingBridges.length > 0`
  - Displays: "Incomplete bridge: X USDC (Chain A -> Chain B)"
  - Resume and Dismiss buttons
  - Hidden during active bridging
- **beforeunload warning**: Prevents accidental tab close during bridging
- Passes `persistenceConfig` via `useMemo` to `useBridge()`
- Integrates `useRecovery()` hook

## Phase 2: Code Review Fixes

### Issues Fixed

#### Critical
1. **Fragile synthetic BridgeResult** (useBridge.ts) - Replaced unsafe `as string` casts with BridgeChain enum values. Moved chain resolution before handlers. Added progressive `bridgeResult.steps` updates in attestation/mint handlers.

#### Warnings
2. **Duplicated isEIP1193Provider** - Extracted to `utils.ts`, removed copies from `useBridge.ts` and `useRecovery.ts`
3. **Object ref in useCallback deps** (useBridge.ts) - Stored `persistence` config in ref, removed from dependency array
4. **Unsafe txHash type assertion** (useRecovery.ts:122) - Replaced `as` cast with `toHexString()` validation
5. **O(n^2) cleanup** (storage.ts) - Batch index update instead of per-record `removePendingBridge` calls
6. **No max record limit** (storage.ts) - Added `MAX_RECORDS = 50` cap with oldest eviction

#### Suggestions
7. **BridgeProgress hides on error** - Now shows with error indicator on failed step
8. **getChainName recreated every render** (RecoveryBanner) - Moved to module-level `getChainNameFromConfigs()`
9. **types.ts import() syntax** - Replaced with proper top-level import
10. **getStepState doesn't handle error** - Added `"error"` state with `lastActiveStep` tracking
